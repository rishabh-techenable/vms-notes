Section 01

Custom Class Training: Principles and Process

Question

YOLO has a pre-trained class for "truck." Only the car and truck classes are required. A new custom class needs to be added using a proprietary dataset. Describe the end-to-end training process — what changes, what improves, and how it works internally.

Before beginning any training procedure, a critical machine learning principle must be addressed: Catastrophic Forgetting. If a pre-trained YOLO model is fine-tuned exclusively on images of a new custom class, it will entirely overwrite its knowledge of car and truck detection. To retain all target classes simultaneously, the model must be trained on a unified dataset containing annotations for all classes together.

Training Process

  1. 1

    Build the combined dataset. Source car and truck images from COCO or OpenImages. Combine with custom class images. Normalize all annotations to YOLO format: class_id x_center y_center width height. Re-index class IDs sequentially: 0: car, 1: truck, 2: custom_class.

  2. 2

    Author the data.yaml file. This file declares the training and validation image paths, the number of classes (nc: 3), and the class name mapping.

  3. 3

    Initialize transfer learning. Load the pre-trained YOLOv8x weights (yolov8x.pt) as the starting checkpoint. Direct the training command to the new data.yaml.

  4. 4

    Execute training. The model iterates over the combined dataset for a defined number of epochs, converging its weights toward the three-class detection problem.

data.yaml
train: /path/to/dataset/train/images
val:   /path/to/dataset/val/images

nc: 3
names: ['car', 'truck', 'custom_class']

What Changes During Training

Output head (architecture) The 80-class output layer is discarded and replaced with a randomly initialized 3-class head.
Final layer weights Undergo large adjustments as the model learns to distinguish the new custom class.
Early layer weights Undergo minor refinement. Edge and texture detectors are largely preserved from pre-training.
Loss calculation The model no longer penalizes missed predictions for classes outside the defined three.

What Improves

  • Domain accuracy: Fewer classes means greater specialization. The model's capacity is concentrated entirely on the target objects.
  • False positive reduction: Removing irrelevant classes eliminates erroneous cross-class predictions.
  • Custom class recall: Leveraging the pre-trained backbone's general visual understanding significantly reduces the labeled data volume required for the new class.

Transfer Learning Mechanics

A YOLO model consists of two principal components. The Backbone functions as the model's perceptual system — trained on millions of images to detect edges, textures, colors, and abstract shapes. This component is preserved. The Head maps abstract feature representations to specific class labels.

During fine-tuning, the backbone's accumulated "visual knowledge" is retained while the head is rewired to output only the three target class predictions.

Section 02

Class ID Architecture and Feature Collision

Question

Currently running YOLOv8x exported to TensorRT, using classes 2 (car) and 7 (truck) from the COCO index. The intent is to add a new class to identify custom tractors under a different name, avoiding ID collision with YOLO's native truck class. Is this approach architecturally sound?

Critical: This approach will produce both a structural and a visual collision. The architecture cannot maintain non-sequential class IDs, and the neural network cannot resolve visual similarity through integer separation alone.

Structural Reality: Sequential IDs Are Mandatory

The YOLOv8x output head is architected to emit probability distributions across exactly 80 contiguous classes (IDs 0–79). There is no mechanism to selectively output IDs 2, 7, and a new ID such as 80. The architecture strictly requires a zero-indexed sequential mapping.

The output head must be rebuilt from scratch. The class remapping for a 3-class model is as follows:

New ID 0 Formerly COCO ID 2 — Car
New ID 1 Formerly COCO ID 7 — Truck
New ID 2 Custom class — Tractor/Proprietary vehicle

Visual Reality: Feature Collision in the Network

The model does not process integer class IDs — it processes pixel gradients, geometric shapes, and learned feature maps. If the custom tractor shares visual characteristics with the standard COCO truck (large tires, elevated cab, exhaust infrastructure), the loss function will generate high penalties during training as the model's pre-trained weights conflict with the new label assignments.

Mitigation: Include labeled images of standard trucks alongside custom tractor images in the training dataset. Perform deliberate hard negative mining on visually ambiguous border-case vehicles to force the network to learn discriminating features between the two classes.

End-to-End Pipeline to TensorRT

  1. 1

    Dataset preparation. Combine COCO car/truck images with custom tractor images. Normalize all annotation IDs to the new 0, 1, 2 scheme.

  2. 2

    PyTorch training. Load yolov8x.pt, point to the new 3-class data.yaml, and train. The 80-class head is discarded; a new 3-class head is initialized and trained.

  3. 3

    ONNX export. Export the best checkpoint weights to an ONNX intermediate graph.

  4. 4

    TensorRT compilation. Compile the ONNX graph into a new .engine file via trtexec. Update the inference pipeline to consume the new output tensor shape with 3 classes.

Section 03

Retraining, Convergence Time, and Continuous Learning

Questions

1. When retraining a deployed model, is the process identical to initial training, or can the new dataset alone be used as input?  2. Will training time be equivalent to or shorter than the initial run?  3. What dataset volume is appropriate when appending data to existing classes?  4. When the deployed model misses detections in production, should retraining wait for a large image batch, or is progressive updating viable?

1. Process: Fine-Tuning vs. Initial Training

The starting checkpoint changes from the generic yolov8x.pt to the previously trained custom best.pt. Training on the new dataset in isolation is not viable — this will cause Catastrophic Forgetting, overwriting the model's knowledge of cars and trucks.

The correct strategy is a Replay Buffer: mix the newly collected images with a random 10–20% sample of the original baseline dataset. Train the current best.pt on this mixed batch.

2. Convergence Time

Retraining is significantly faster. The initial run required the classification head to reorganize from scratch, typically demanding 100+ epochs. Subsequent fine-tuning merely nudges existing weights to accommodate edge cases, generally converging within 10–30 epochs.

3. Dataset Volume for Class Updates

Guidance: Small and highly curated is preferable to large and redundant. A batch of 50–200 high-diversity, difficult images will yield greater accuracy improvement than 5,000 easy, repetitive images. Prioritize varied lighting, angles, occlusions, and confuser vehicles.

Maintain a balanced class ratio across the mixed training set to prevent the model from becoming biased toward the most recently added class.

4. Progressive Updating via Active Learning

The modern production standard is Active Learning — a continuous cycle of extracting failure frames, annotating them, and triggering micro-updates. This is superior to waiting for a large batch.

  1. 1

    Identify the miss. When the inference pipeline fails to detect or misclassifies an object, extract the specific frame from the rolling video retention.

  2. 2

    Annotate the frame. Label the object correctly in the extracted frame.

  3. 3

    Stage the frame. Drop it into a designated staging directory.

  4. 4

    Trigger micro-updates. Once the staging batch reaches 50–100 images, run a short fine-tuning pass on the current weights, export to ONNX, rebuild the TensorRT engine, and swap it into the inference pipeline.

Section 04

Understanding Weights and TensorRT Optimization

Question

How do neural network weights function mechanically, and what changes during each training cycle? Additionally, what training and export decisions will improve the quality and performance of the final TensorRT engine file?

How Weights Function

A weight is a numerical multiplier that encodes the relative importance of a specific visual input signal. The function of these multipliers varies by network depth:

Early layers Act as basic visual filters — detecting edges, gradients, and contrast transitions.
Middle layers Combine low-level features into higher-order patterns: textures, curves, geometric shapes (e.g., the circular profile of a tractor tire).
Final layers (Head) Map abstract feature combinations to class probabilities (e.g., "large tire + elevated cab + exhaust stack = 92% Class 2: Tractor").

The Training Update Cycle

Every training batch executes three steps in sequence:

  1. 1

    Forward pass. An image traverses the network. Current weights multiply pixel values and features to produce a class prediction.

  2. 2

    Loss calculation. The prediction is compared against the ground-truth annotation. The mathematical difference — the "loss" — is computed.

  3. 3

    Backpropagation. Using calculus, the system traces which specific weights contributed to the incorrect prediction and incrementally adjusts those values to reduce future error.

Optimizing the Training Process for TensorRT

  • Lock the input resolution: Train at the exact inference resolution (e.g., imgsz=640). Export with fixed input dimensions rather than dynamic shapes wherever possible for maximum kernel optimization.
  • Avoid custom operations: Stick to native YOLOv8 architecture. Non-standard operations may not be fusable by TensorRT's kernel compiler, creating inference bottlenecks.
  • Prepare an INT8 calibration dataset: If INT8 quantization is planned, reserve a representative set of 500–1,000 training images to feed the TensorRT calibration process during engine build.

Clean ONNX Export

TensorRT cannot parse .pt files directly. The ONNX intermediate format is required. The simplify=True flag is mandatory — it runs a graph optimizer to eliminate redundant operations and constant-fold mathematical expressions before TensorRT compilation.

Export command
yolo export model=runs/detect/train/weights/best.pt \
  format=onnx opset=12 simplify=True
Section 05

Dynamic Resolution Engine Compilation

Question

Multiple RTSP feeds from different camera models require dynamic input resolution support. How should the ONNX export and TensorRT compilation be configured for a dynamic engine?

Step 1 — Dynamic ONNX Export

The dynamic=True flag instructs PyTorch to leave batch size and spatial dimensions as variable axes in the exported ONNX graph.

ONNX export (dynamic)
yolo export model=best.pt format=onnx dynamic=True simplify=True

Step 2 — TensorRT Optimization Profile

A dynamic ONNX model requires an explicit Optimization Profile during TensorRT compilation. Three shape boundaries must be defined (format: batch × channels × height × width):

--minShapes The minimum input dimensions the engine will ever receive.
--optShapes The most common production dimensions. TensorRT optimizes its execution kernels most aggressively for this size.
--maxShapes The maximum dimensions. GPU memory is allocated based on this value.
trtexec — dynamic engine compilation
trtexec --onnx=best.onnx \
  --saveEngine=best_dynamic.engine \
  --minShapes=images:1x3x320x320 \
  --optShapes=images:1x3x640x640 \
  --maxShapes=images:4x3x1080x1920 \
  --fp16
VRAM caution for Jetson Orin Nano: GPU memory is pre-allocated at the --maxShapes boundary. Inflating this value unnecessarily will consume shared memory and risk OOM crashes in the GStreamer pipeline. Set --maxShapes as tightly as actual RTSP stream resolutions permit.
Section 06

Dataset Construction and Annotation Workflow

Question

Describe the complete process for creating datasets for new custom classes and updating annotations for existing classes.

Phase 1 — Data Sourcing (Hard Mining from Production NVR)

  1. 1

    Isolate failures. Note timestamps where the current pipeline misses a vehicle or produces a misclassification.

  2. 2

    Extract sparse frames using FFmpeg. Pull 5–10 second clips around failure timestamps. Extract at 1–2 fps to ensure visual diversity between frames. Extracting at 30 fps yields 300 near-identical images per clip, which causes overfitting.

  3. 3

    Gather confuser frames. Specifically extract standard trucks that resemble the custom tractor, and occluded or poorly lit cars. These become training hard negatives.

Phase 2 — Annotation

Recommended tooling:

CVAT

Open-source, self-hosted via Docker. Industry standard. Native YOLO export. Recommended for production data pipelines.

Label Studio

Open-source, self-hosted. Strong API integration support for automated pipelines.

Roboflow

Cloud-based, polished UI. Excellent dataset versioning. Has tiered pricing.

Labeling discipline: Bounding boxes must be drawn as tightly as possible around visible vehicle pixels only. Define clear, written criteria distinguishing each class before annotation begins. Annotate only what is visible; never infer occluded geometry.

Phase 3 — Replay Buffer Construction

  1. 1

    Draw a random 15–20% sample from the original baseline dataset (cars and trucks).

  2. 2

    Include 100% of the newly annotated NVR frames.

  3. 3

    Merge into a single combined dataset for training.

Phase 4 — Directory Structure

Required directory layout
custom_dataset/
├── data.yaml
├── images/
│   ├── train/     ← 80% of all images
│   └── val/       ← 20% of all images
└── labels/
    ├── train/     ← Corresponding .txt annotation files
    └── val/       ← Corresponding .txt annotation files
Validation set rule: The val/ directory is never exposed to the model during training. It must contain a representative mix of all three classes, including edge cases, to produce honest performance metrics.
Section 07

Frame Extraction Strategies from Live Video

Question

Available source material consists of 30fps H.264/H.265 MP4 event clips where the target vehicle is moving throughout the recording. The current YOLOv8x model detects the vehicle inconsistently — identifying it in some frames and missing it in similar adjacent frames. How should frame extraction for annotation be approached?

Method 1 — FFmpeg Frame Decimation

Suitable for rapid batch extraction. Downsample the video to 1–2 fps to ensure sufficient vehicle movement between extracted frames, guaranteeing visual diversity.

FFmpeg — 1 frame per second extraction
ffmpeg -i input_event.mp4 \
  -vf "fps=1" \
  -q:v 2 \
  extracted_frames/frame_%04d.jpg

The -q:v 2 flag maintains high JPEG quality. High compression artifacts in training data degrade model accuracy.

Method 2 — Active Confidence-Based Extraction (Recommended)

Leverages the deployed model to automatically isolate the exact frames where detection fails. This eliminates manual frame hunting and produces maximally difficult training examples.

Extraction logic: Save a frame if there are zero detections in the expected region, or if the maximum detection confidence falls within a defined "gray zone" (e.g., 0.25–0.50). Apply a cooldown counter (e.g., skip 15 frames after each save) to prevent saving redundant near-identical frames during sustained failures.

Python — active confidence extraction
import cv2
from ultralytics import YOLO

model = YOLO("yolov8x.pt")
cap = cv2.VideoCapture("input_event.mp4")

frame_count = 0
cooldown = 0

while cap.isOpened():
    ret, frame = cap.read()
    if not ret:
        break

    if cooldown > 0:
        cooldown -= 1
        frame_count += 1
        continue

    results = model(frame, classes=[2, 7], verbose=False)

    if len(results[0].boxes) == 0:
        cv2.imwrite(f"hard_negatives/missed_{frame_count}.jpg", frame)
        cooldown = 15
    else:
        conf = float(results[0].boxes.conf[0])
        if conf < 0.50:
            cv2.imwrite(f"hard_negatives/low_conf_{frame_count}.jpg", frame)
            cooldown = 15

    frame_count += 1

cap.release()

Annotation Guidelines for Extracted Failure Frames

Motion blur If a human can reasonably identify the vehicle despite blur, draw the bounding box. The model must learn blurred feature representations.
Partial occlusion Draw the bounding box strictly around the visible portion. Never extend the box to infer hidden geometry.
Extremely small objects Skip frames where the vehicle is fewer than ~10 pixels on any axis. Such labels introduce noise into the loss function.
Section 08

Advanced Annotation Methodology

Question

What advanced annotation practices should be followed for a production VMS system that relies on bounding box quality for both classification accuracy and downstream tracker (DeepSORT / ByteTrack) ID consistency?

Precision for Tracking and ROI Logic

  • Pixel-perfect edges: Bounding boxes must touch the absolute outermost visible pixels. Loose boxes cause false IoU calculations and premature tracker ID drops.
  • Exclude protrusions: Do not include antennas, thin side mirrors, or exhaust pipes. These inflate box dimensions and create overlap artifacts in dense traffic.
  • Temporal stability: For a vehicle approaching the camera, annotated boxes should grow consistently frame over frame. Jittery boxes will corrupt velocity estimation in tracker logic.

Edge Cases in Long-Duration NVR Recordings

Nighttime headlight glare Box only the physical headlight hardware and visible chassis. Do not include road illumination.
Frame-edge truncation Draw the box around the visible portion with the edge of the box flush with the image boundary.
Articulated vehicles (tractor-trailer) Establish and document a strict rule: either label the cab only (recommended for ROI line-crossing accuracy) or the full unit. Do not mix approaches within the dataset.
Wet road reflections Box the physical vehicle only, never the road reflection.
Cast shadows Never include road shadows within the bounding box boundary.

Class Definition Consistency

Create a Visual Heuristic Guide: a reference sheet containing 5 canonical screenshots per class with written criteria defining the distinguishing visual characteristics (e.g., "Tractor: exposed rear wheels, open frame chassis; Truck: enclosed box bed, fully paneled body"). Any ambiguous vehicle that requires more than 3 seconds of consideration should be discarded from the dataset rather than labeled with uncertainty.

Efficient CVAT Annotation Workflow

  1. 1

    Bind hotkeys: Map class IDs to numeric keys (1 = Car, 2 = Truck, 3 = Tractor). Map N to create a new bounding box, F to advance frame.

  2. 2

    First pass: Rapidly draw approximate boxes around all target vehicles, assign class labels, advance frame.

  3. 3

    Refinement pass: Review the batch and tighten all boxes to pixel-precise boundaries. Two-pass annotation is significantly faster than aiming for perfection on first draw.

  4. 4

    Post-export verification: Open a sample of exported .txt files and confirm the first integer on each line matches the expected class ID mapping before ingesting into training.

Section 09

Hard Negative Mining: Theory and Practice

Question

What is a "hard negative" in the context of object detection, and how does it differ from a standard negative example?

Definition

A hard negative is an image region that does not contain the target object but is visually similar enough to the target that the model incorrectly predicts a high-confidence match. It is "negative" because it is not the target class; it is "hard" because it systematically deceives the model's current decision boundary.

Easy vs. Hard Negatives

Easy negative

A motorcycle or a clear roadway when scanning for tractors. The model resolves this correctly with near-zero loss contribution. Weights do not update meaningfully.

Hard negative

A heavy dump truck or backhoe sharing large-tire and elevated-cab geometry with the target tractor. The model may confidently misclassify this, generating large loss and forcing significant weight updates.

Effect on training

Hard negatives drive targeted, high-magnitude weight updates, forcing the model to learn fine-grained discriminating features rather than broad shape heuristics.

Hard Negative Mining in Production

Hard negative mining is the practice of systematically identifying these deceptive examples from the production environment and incorporating them into the training dataset. Rather than adding thousands of generic tractor images (which reinforce knowledge the model already has), 100 images of the specific standard trucks that the model falsely flags as tractors will force the network to learn the precise visual boundary between the two classes, directly reducing the false positive rate after retraining.

Section 10

Model Version Control and MLOps

Question

As the model will be continuously retrained in production, how should model versions, datasets, experiment results, and edge deployment configurations be systematically managed to prevent regression and data loss?

Semantic Versioning Nomenclature

Every artifact must be named with a schema that fully describes its contents without opening the file.

Version tier Trigger condition Example
Major (v2.0.0) Architecture changes or class schema changes (e.g., adding a new class, upgrading to a new YOLO version) YOLOv8 → YOLOv11
Minor (v2.1.0) Significant dataset additions or replay buffer retraining rounds +500 hard negative frames
Patch (v2.1.1) Hyperparameter adjustments on identical dataset Learning rate tuning

File naming standard: Every artifact includes version, architecture, resolution, precision, and target hardware.
Example: trinetra_v2.1.0_yolov8x_img640_fp16_orin.engine

Data Version Control (DVC)

Git is not suitable for dataset management — binary image files will corrupt repository performance. DVC (Data Version Control) functions identically to Git but stores actual image data in a remote object store (AWS S3, local NAS) and commits only a lightweight pointer file to the Git repository. This permanently locks a code version to its corresponding dataset state, enabling complete experiment reproducibility.

Experiment Tracking

Weights & Biases (W&B) integrates natively with YOLOv8 and automatically logs per-epoch loss, mAP curves, and validation sample images to a cloud dashboard. This creates a permanent, queryable ledger of every training run, enabling direct comparison of accuracy metrics across model versions.

W&B setup
pip install wandb
wandb login

Edge Deployment Configuration Registry

Model paths must not be hardcoded in inference scripts. A config.yaml file, read by the pipeline on startup, acts as the central source of truth for the active model version. Rollback to any previous version requires only a YAML edit and a service restart.

config.yaml — deployment manifest
system:
  camera_feed: "rtsp://..."
  target_fps: 30

model:
  version: "v2.1.0"
  architecture: "yolov8x"
  precision: "fp16"
  engine_path: "/models/trinetra_v2.1.0_yolov8x_img640_fp16_orin.engine"
  classes:
    0: "car"
    1: "truck"
    2: "tractor"
Section 11

On-Premise Model Delivery and OTA Updates

Question

What are the recommended strategies for shipping updated models to on-premise Jetson devices? Compiling a TensorRT engine on the production Jetson while inference is running is not viable due to memory pressure. Consider an OTA-style approach where devices periodically check a cloud registry and build the engine during a maintenance window, or receive a pre-compiled engine file.

Critical constraint: trtexec is resource-intensive and will spike CPU utilization, exhaust shared memory, and cause OOM crashes in active inference pipelines. Engine compilation must not occur on production hardware during live inference. Additionally, .pt PyTorch files must never be shipped to edge devices — the PyTorch runtime is excessively large for edge containers. Ship .onnx or pre-compiled .engine files only.

Strategy Comparison

Strategy 1 — Scheduled Nightly Build on Production Hardware

Viable with caveats

A cron job polls a cloud or local manifest for new model versions. On detection, the ONNX file is downloaded to a staging directory. At a defined maintenance window (e.g., 03:00), the inference service is halted, trtexec builds the engine from the staged ONNX, the active model symlink is updated, and the service is restarted.

Advantages

Self-contained. No additional hardware required.

Disadvantages

Requires 5–15 minutes of planned VMS downtime during compilation.

Strategy 2 — Shadow CI Build Runner (Recommended)

Zero production risk

A dedicated "Shadow" Jetson Orin Nano (identical hardware to the production units) is maintained offline from the inference pipeline. When a new ONNX file is published to the registry, the Shadow Jetson downloads it, compiles the engine via trtexec, and uploads the finished .engine file back to the registry. Production Jetsons poll the manifest, download the pre-compiled engine, and perform a 5-second service restart to load the new weights — zero compilation time on production hardware.

Note: TensorRT engine files are platform-specific. An engine compiled on an x86 RTX machine cannot run on an ARM Jetson. The Shadow Jetson must be identical hardware to the deployment targets.

Advantages

Production inference downtime reduced to a 5-second restart. Zero compilation risk on live hardware.

Disadvantages

Requires one additional Jetson unit dedicated to CI builds.

Strategy 3 — Local MinIO Registry (Air-Gapped Environments)

Air-gap compatible

For sites with no reliable internet connectivity, a MinIO container (S3-compatible object store) runs on the primary local server. Edge devices poll this local registry. This approach is combined with either Strategy 1 or 2 for the compilation step.

Advantages

No internet dependency. Full data sovereignty. Low latency on local network.

Disadvantages

Requires maintaining a local server with persistent storage.

OTA Poller Implementation

ota_poller.py — runs via cron
import os, requests, subprocess, json

MANIFEST_URL = "http://your-local-minio-or-s3/models/manifest.json"
LOCAL_MANIFEST = "/opt/trinetra/models/current_manifest.json"
STAGING_DIR = "/opt/trinetra/models/staging/"

def check_for_updates():
    try:
        remote = requests.get(MANIFEST_URL).json()
        with open(LOCAL_MANIFEST, 'r') as f:
            local = json.load(f)
        if remote["version"] != local["version"]:
            download_model(remote)
    except Exception as e:
        print(f"Update check failed: {e}")

def download_model(manifest):
    path = os.path.join(STAGING_DIR, "new_model.engine")
    r = requests.get(manifest["engine_url"], stream=True)
    with open(path, 'wb') as f:
        for chunk in r.iter_content(chunk_size=8192):
            f.write(chunk)
    subprocess.run(["/bin/bash", "/opt/trinetra/scripts/deploy_model.sh",
                    manifest["version"]])

if __name__ == "__main__":
    check_for_updates()
deploy_model.sh — atomic deployment
#!/bin/bash
NEW_VERSION=$1
STAGING="/opt/trinetra/models/staging/new_model.engine"
PROD_DIR="/opt/trinetra/models/production"
SYMLINK="/opt/trinetra/models/active.engine"

# Guard: abort if staged file is empty
if [ ! -s "$STAGING" ]; then
    echo "Deployment aborted: staged model file is missing or empty."
    exit 1
fi

# Move to versioned production path
mv "$STAGING" "$PROD_DIR/trinetra_${NEW_VERSION}.engine"

# Atomic symlink swap
ln -sf "$PROD_DIR/trinetra_${NEW_VERSION}.engine" "$SYMLINK"

# Restart inference service
systemctl restart trinetra-inference.service

# Update local version state
echo "{\"version\": \"$NEW_VERSION\"}" > /opt/trinetra/models/current_manifest.json
echo "Deployment complete: $NEW_VERSION"
